# Icon And Component Systems

Use this for icon systems, glyph sets, visual component libraries, symbol grids, UI kits, component cards, and system marketing.

## Core Feel

Icons and components should feel authored as one family.

- Consistent stroke, fill, radius, and optical weight.
- Shared grid and alignment rules.
- Clear metaphors.
- Repeated sizing and padding.
- Strong negative space.
- Labels only when they improve scanning.

## Icon Rules

- Pick outline, filled, duotone, or solid. Do not mix casually.
- Use one stroke width per family.
- Align icons optically, not only mathematically.
- Keep corner style consistent.
- Use `currentColor` for UI icons unless brand color is required.
- Avoid generic icon swaps when the metaphor matters.

## Component Library Cards

For visual systems or component cards:

- Show a real component state: default, active, hover, disabled, loading, error.
- Include compact labels, tokens, or specs only when useful.
- Use a restrained grid or table for comparison.
- Show repeated variants as a family.
- Use enough whitespace that the system feels premium.

## Symbol Grids

- Use a stable grid.
- Keep symbol count manageable.
- Group related symbols.
- Do not let the grid overpower the actual icons.
- Use contrast to show selected or primary examples.

## Product Marketing Use

Icon/component systems can support:

- Integration grids.
- Feature taxonomy.
- Design-system pages.
- Developer handoff.
- Capability cards.
- Navigation and toolbar previews.

Avoid using icon grids as filler.

## QA

- Do icons look like one family?
- Are metaphors specific and understandable?
- Are states represented clearly?
- Are icons aligned and optically balanced?
- Would the system still work at small sizes?
